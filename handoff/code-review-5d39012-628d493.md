## Strengths

The work in this range is, on the whole, careful and well-defended. Specific things done well:

- **Auth change is minimal and correctly reasoned.** `core/server-auth.ts:116` flips the guard from "local-only" to "deny only custom_remote." This is the right call: the server's `connectionKind` is derived from the configured `networkMode` + host/remote-address checks in `server/http/transport-context.ts` (wired at `server/index.ts:374-383`), not from any client-supplied value, so an attacker cannot smuggle a `lan` classification to unlock query-token auth. The `loopback_token` is still hard-rejected for non-local transport at `server-auth.ts:64`, so the change only broadens acceptance of *device credentials* over LAN query strings — which is exactly what the WebSocket auth fix needed. `principalAllowsConnection` (`server-auth.ts:136-143`) keeps the trust-state matrix intact.
- **Plugin iframe credential leak fix is correctly scoped.** `desktop/src/react/hooks/use-plugin-surface-url.ts:131` gates `includeTokenQuery` on `isLocalOwnerConnection(connection)`, so remote/tunnel plugin iframes no longer carry the device credential in their URL. Combined with `canUseQueryToken` rejecting `custom_remote` (`server-connection.ts:165`), remote iframe URLs are now credential-free. The test at `server-connection.test.ts` ("does not put remote device credentials into browser-loadable URL query strings") locks this in.
- **CSP widening is tight and tested.** `connection-csp.js` adds a new `media-src` directive and only injects *http(s) origins* (via `httpSourcesOnly`) into `img-src`/`media-src`, never bare `http:`/`https:` schemes. `tests/csp-sync.test.ts` asserts both the inclusion of the specific LAN origin and the absence of wildcard schemes — exactly the invariant CLAUDE.md demands.
- **`connect:probe` SSRF hardening is thoughtful.** `desktop/main.cjs:4569-4604` validates the scheme (`^https?://`), validates the sender frame is `file://` (Electron security), runs in main via `net.fetch` (CSP-immune), and swallows network errors into `{ok:false}` rather than rejecting through `wrapIpcHandler`'s error path. The chicken-and-egg rationale (CSP built from active connection) is documented inline.
- **Install-server checksum hardening is real.** `628d493c` replaced an `sh -c "printf ... | sha256sum -c"` invocation (which interpolated the asset name into a shell string) with a direct `sha256sum [file]` call plus an explicit `actual !== expected` comparison, and added `safeAssetArchiveName()` to reject path separators/reserved names. The dedicated test ("verifies checksums without interpolating release asset names into a shell") uses a hostile name containing `'; touch pwn #` and asserts no `sh -c` is ever invoked. This is exactly how to close a shell-injection surface.
- **Upload materialization is fail-safe and symlink-aware.** `core/desktop-session-submit.ts:513-538` copies (not moves) temp uploads into the session-files cache, refuses symlinks (`stat.isSymbolicLink()`), uses `COPYFILE_EXCL`/`errorOnExist` to avoid clobbering, bounds the uniqueness loop, and falls back to the original path on any error so submission never breaks. The test at `desktop-session-submit.test.ts` ("materializes temp upload display attachments into the session file cache") verifies the real file ends up under `session-files/` and the original upload path is no longer referenced.
- **Auto-update disabling is layered correctly.** `build-info.cjs` is the single source of truth; `auto-updater.cjs` checks `isUpdateEnabledForBuild()` at every entry point (init, auto-check, manual check, channel set, download no-op) and short-circuits to a `disabled` state. `AboutTab.tsx` hides the controls when `updatesEnabled` is false. The test at `auto-updater.test.ts` confirms `setFeedURL`/`checkForUpdates` are never called for a disabled build.
- **Fork tooling is genuinely useful and self-tested.** `sync-upstream.mjs` and `track-upstream-issues.mjs` never auto-submit upstream issues (the `ISSUE_COMMANDS` array is `["status","search","draft"]`), redact credential examples in drafts, and filter fuzzy GitHub matches. `FORK_SYNC.md` is an unusually thorough divergence runbook with per-file risk ratings.

## Issues

### Critical (Must Fix)

None found. The security-sensitive paths examined (auth token handling, filename sanitization, credential leak prevention, shell injection in tooling) are all sound. No data-loss, broken-functionality, or exploitable bugs identified.

### Important (Should Fix)

**1. `connect:probe` SSRF surface is wider than the comment implies — `net.fetch` follows redirects.**
`desktop/main.cjs:4590-4604`. The handler validates the *initial* URL scheme, but Electron's `net.fetch` follows HTTP 3xx redirects by default. A compromised renderer (or the legitimate use case hitting a misconfigured server) could be steered from an allowed `https://attacker` to an internal `http://169.254.169.254/...` (cloud metadata) or an internal admin endpoint, and the response body of `/api/server/identity` is returned to the renderer (`return { ok: true, identity }`). The blast radius is limited because the attacker already controls the renderer and the credential is sent only to the *initial* host (redirect targets don't receive the Authorization header automatically here since the login POST sends the credential in the body and the identity GET uses an explicit header that `fetch` *does* forward on same-origin redirects but strips on cross-origin), but this is still a server-side request forgery primitive that can probe internal HTTP services and exfiltrate status codes / error strings.
*Why it matters:* The inline comment claims the SSRF guard is "only allow http/https URLs," but scheme-allowing is not origin-pinning. This is the one new main-process network primitive in the range.
*Fix:* Either (a) pass `{ redirect: "manual" }` and reject any non-3xx or cross-origin redirect, or (b) resolve and pin the host before fetching and reject if it resolves to a private/loopback range unless it matches the currently-configured LAN server. At minimum, do not return the response body to the renderer on the identity fetch — return only `{ ok: true }` and let the renderer re-fetch after reload.

**2. No `Referrer-Policy` set, and LAN device-credential tokens now flow through query strings.**
`canUseQueryToken` (`server-connection.ts:162-167`) now returns true for `lan` + `device_credential`, so the device credential is appended as `?token=...` to resource/content and WS URLs for LAN connections (confirmed by the test "appends query token to WebSocket URL for LAN device-credential connections"). The repo sets no `Referrer-Policy` anywhere (grep found none in `desktop/src/*.html`, `vite.config.ts`, or `server/index.ts`). If a remote/LAN-rendered page ever loads a third-party resource or the user clicks an external link, the `Referer` header will leak the credential-bearing URL. For the local loopback this was always true but low-risk; for a LAN device credential that is a real shared secret, the exposure is more serious.
*Why it matters:* A device credential in a `Referer` header to any third party is a credential leak — the exact class of bug this range is trying to close for plugin iframes.
*Fix:* Add `Referrer-Policy: no-referrer` (or `same-origin`) as a `<meta>` in the renderer HTML and/or a response header from the server, and confirm `buildConnectionUrl` consumers don't navigate top-level to external origins with the token attached. This is partially pre-existing but is *materially widened* by this range, so it belongs here rather than as a pre-existing note.

**3. `assertNoDestructiveDataSteps` gives false confidence — it only scans the plan's display strings.**
`scripts/install-server.mjs:430-435`. The regex scans `step.command` text fields for `rm -rf|delete data|...`. But the *actual* shell ops in `createShellUpgradeOps` (the `backup`/`download`/`extract` functions) are what run on `--execute`, and those aren't validated by this check. The `tar -czf` backup includes `plan.paths.currentLink` (a symlink) and `configDir`/`dataDir` — fine — but nothing structurally prevents a future op from running `rm -rf`. The guard is security theater against the display layer, not the execution layer.
*Why it matters:* The reinit-data docs (`docs/reinit-data-failsafe.md`) lean on this as a safety invariant ("fail-closed until backup/restore gates are implemented"). A reader could assume destructive commands are structurally blocked when they're only blocked from appearing in the human-readable plan text.
*Fix:* Either rename to clarify it validates plan *descriptions* only, or add an allowlist of permitted command prefixes in `createShellUpgradeOps.checked` and reject anything outside it. The current `--execute` path already requires all ops to be injected, so tightening the execution boundary is straightforward.

**4. `executeUpgradePlan` rollback is silently skipped if `switchCurrent` itself throws.**
`scripts/install-server.mjs:478-498`. `switchedCurrent` is only set to `true` *after* `ops.switchCurrent` resolves. If `ln -sfn` partially succeeds then fails (e.g., the symlink was created but the target is invalid, or the operation throws after mutating), `switchedCurrent` stays `false` and rollback won't fire, leaving the service pointing at a broken release on the next `systemctl restart`. The current-link mutation is the one step where partial failure is most dangerous.
*Why it matters:* The rollback contract documented in the test ("rolls back to the previous release when health verification fails after restart") only covers the post-switch health-check path, not a failed switch.
*Fix:* Either make `switchCurrent` atomic by writing to a temp symlink and `rename(2)` over the target, or set `switchedCurrent = true` optimistically inside a try around the symlink swap so rollback fires on switch failure too.

### Minor (Nice to Have)

**5. `materializeTempUploadForSession` TOCTOU between `existsSync` and copy.**
`core/desktop-session-submit.ts:558-565`. `uniqueMaterializedPath` checks `existsSync` then the caller copies with `COPYFILE_EXCL`/`errorOnExist`, so the *security* property holds (no overwrite). But under concurrency the loop could exhaust all 100 candidates and fall through to the `Date.now()` name. Submission is serialized per-session so this is theoretical, but worth a one-line comment noting that `EXCL` is the real guard.

**6. `MAX_ATTACHMENT_BLOB_BYTES` check is partially redundant with the base64-length check above it.**
`server/routes/upload.ts:510-522`. `isUploadBlobBase64WithinLimit` already bounds the base64 char count to `MAX_ATTACHMENT_BLOB_BASE64_CHARS` (25 MiB worth), and then a second check re-validates the decoded `buf.length` against the same 25 MiB. The decoded check is the correct one (base64 length is an over-estimate), so the base64 check is now redundant for attachments — but harmless. Consider a comment explaining both exist intentionally (char-limit fast-path, byte-limit authoritative).

**7. `connection-csp.js` `httpSourcesOnly` regex allows `http://` and `https://` but the CSP `connect-src` does not filter.**
`connection-csp.js:53-58`. `connectSources` includes *all* scoped sources (keys of `scopedSources`), while `img-src`/`media-src` are filtered to http(s) only. The scoped sources for a connection are always http(s)/ws(s) origins anyway (validated upstream), so this is fine today, but the asymmetry is a latent footgun if a non-http source ever enters `scopedSources`. Worth a comment tying the filter to the img/media directive's stricter requirement.

**8. `track-upstream-issues.mjs` and `sync-upstream.mjs` shell out via `spawnSync` with `shell:false` (good) but `commandExists` uses `sh -c "command -v ${command}"`.**
`scripts/sync-upstream.mjs:55`. The `command` values are all hard-coded literals (`git`, `node`, `npx`, `gh`), so there's no injection here, but the pattern is inconsistent with the otherwise-no-shell stance. Cosmetic only.

**9. `BuildInfo` type duplication between `build-info.cjs` (plain JS) and `desktop/src/react/types.ts`.**
The runtime shape is normalized in `build-info.cjs` and re-declared as a TS interface in `types.ts`. They can drift. A code-gen step or a single shared schema would be more robust, though for an 8-field object this is minor.

## Recommendations

- **Pin the `connect:probe` network boundary** (Important #1). This is the only genuinely new main-process attack surface in the range and it's the one place I'd want a second pair of eyes before relying on it. Consider restricting the probe to URLs whose host matches a server already in the persisted connection registry, plus `redirect: "manual"`.
- **Add a `Referrer-Policy` header/meta globally** (Important #2). It's a one-line, defense-in-depth fix that materially protects the new LAN-query-token path and costs nothing.
- **Consider an integration test for the end-to-end LAN preview invariant.** The current tests verify the URL-building and CSP pieces in isolation, but CLAUDE.md's bar is "both chat thumbnails AND Conversation Files previews still render" after switch-chat/return. The unit coverage is strong; a single Vitest that drives `resolveFileRefUrl` through the `user-attachment-media` → `AttachedFilesBar` path for a remote LAN attachment (no base64, only `fileId`) would catch regressions in the synthetic `res_sf_` path that the per-function tests might miss if the wiring changes.
- **Document the `assertNoDestructiveDataSteps` scope** (Important #3) so the reinit-data failsafe doc doesn't over-claim.
- The `FORK_SYNC.md` divergence table is excellent; keep it updated as part of the Definition of Done for any future security-touch in these files, since it's now the canonical merge-conflict guide.
