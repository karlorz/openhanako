# Reinit Data Failsafe Design

Status: non-mutating dry-run planner implemented in `scripts/install-server.mjs`.
Confirmed reset/import and restore remain unavailable; no destructive behavior
is implemented by this document.

## Goal

Define `install-server reinit-data` as a separate, explicit, recoverable reset/import workflow for first-run or new-user testing. It must never be part of `install-server upgrade`.

The default result is a fresh workspace. The command preserves or imports only the narrow operational settings required to reconnect a test client:

- Providers/LLM settings.
- Access & Devices / Connected Remote Hana settings.

Persona, agents, memories, sessions, workspaces, plugins, user files, and broad customizations are not preserved by default.

## Command Shape

```sh
install-server reinit-data --dry-run
install-server reinit-data --confirm <plan-id>
install-server reinit-data --restore <backup-path>
```

Required behavior:

- `--dry-run` is the default mode when neither `--confirm` nor `--restore` is present.
- `--confirm <plan-id>` must reference a plan generated by a prior dry run.
- Confirmation must be impossible if the plan has expired, the data root changed, or the backup cannot be verified.
- `--restore <backup-path>` restores from a verified backup and does not run reset/import steps.

Current implementation boundary:

- `reinit-data --dry-run` writes a plan file and does not mutate application data.
- `reinit-data --confirm` fails closed until backup, import, reset, audit, and health gates are implemented.
- `reinit-data --restore` fails closed until backup verification and restore operations are implemented.

## Data Root

The implementation must resolve the active Hana data root the same way the server does:

- `HANA_HOME` when set.
- Otherwise the application default, currently `~/.hanako` for CLI/server flows.

The command must print the resolved data root in every dry run and confirmation prompt. If the server is running, status output should identify the active process and service before planning destructive steps.

## Built-In Mechanism Discovery

Before implementing custom file deletion, the implementation work item must inspect and prefer built-in Hana mechanisms:

- CLI commands.
- HTTP/server routes.
- App export/import flows.
- Reset/onboarding flows.
- Existing auth/preferences/provider stores.

If a built-in export/import or reset path exists and can be tested, use it. Custom deletion is allowed only after documenting why no suitable built-in path exists for the exact data class.

## Dry-Run Plan

`install-server reinit-data --dry-run` creates a plan file under an operator-visible plan directory, for example:

```text
/var/lib/hanaagent/reinit-plans/<timestamp>-<short-hash>.json
```

The plan must include:

- Plan id.
- Created time and expiry.
- Resolved data root.
- Service name and current service state.
- Files/directories to back up.
- Files/directories that would be removed or replaced.
- Settings to export/import, with explicit field categories.
- Backup destination.
- Restore command.
- Health checks to run after reset.
- A "not preserved" section naming excluded classes.

The dry run must not mutate application data except for writing the plan file itself.

## Backup Gate

No destructive step may run before a verified backup exists.

Backup requirements:

1. Stop or quiesce the service when needed to get a consistent snapshot.
2. Archive the entire resolved data root plus relevant config files.
3. Include release/service metadata and the dry-run plan.
4. Write a manifest with file list, sizes, and SHA-256 checksums.
5. Verify the archive by reading it back and checking the manifest.
6. Print the backup path and restore command.

If backup creation or verification fails, the command exits before reset. It must not partially clear data.

## Export Scope

The export/import scope is narrow by default.

Allowed export categories:

- Provider credentials and model/provider preferences needed to run LLM calls.
- OAuth/provider custom model settings when they are part of provider setup.
- Access/device connection settings needed for Connected Remote Hana test setup.
- Server network/access configuration required to reconnect the desktop client.

Excluded by default:

- Persona.
- Agents.
- Memories.
- Session history.
- Workspaces and desk content.
- Uploaded files and generated media.
- Plugins and plugin state.
- General preferences unrelated to remote connection or provider setup.

The implementation must map these categories to concrete files/routes/stores during implementation and add tests for every included path.

## Reset Sequence

The confirmation flow should be:

1. Load and validate the dry-run plan.
2. Re-resolve the data root and refuse if it differs from the plan.
3. Verify the plan is not expired.
4. Stop or quiesce `hanaagent`.
5. Create and verify the failsafe backup.
6. Export the narrow allowed settings from the backup or live data root.
7. Reset to a fresh data root using built-in reset behavior when available.
8. Import only the allowed settings.
9. Start `hanaagent`.
10. Run health checks.
11. Write an audit report next to the backup and plan.

If health checks fail after reset, the command prints the restore command and may auto-restore only if it can prove no new user data was created after reset. Otherwise it stops and leaves the verified backup untouched for operator restore.

## Restore Sequence

`install-server reinit-data --restore <backup-path>`:

1. Verifies the backup archive and manifest.
2. Stops or quiesces `hanaagent`.
3. Moves the current data root aside with a timestamp; do not delete it immediately.
4. Restores the backup data root and config files.
5. Fixes ownership and permissions.
6. Starts `hanaagent`.
7. Runs health checks.
8. Prints the restored version, data root, and retained aside path.

Restore must be available even if a previous reinit run failed halfway.

## Audit Output

Every confirmed run writes an audit file containing:

- Operator command.
- Plan id.
- Backup path.
- Data root.
- Exported categories.
- Removed/replaced paths.
- Health check result.
- Restore command.
- Start/end timestamps.

The audit output must avoid printing secret values. It may print key names, provider ids, model ids, and redacted credential presence.

## Failure Policy

- Missing plan id: show dry-run instructions and exit nonzero.
- Plan expired: require a fresh dry run.
- Data root mismatch: refuse to continue.
- Backup verification failure: refuse to reset.
- Export failure for allowed settings: refuse to reset unless the operator explicitly chooses fresh-only mode in a future design.
- Reset failure before import: print restore command.
- Import failure: keep backup and print both restore and retry instructions.
- Health check failure: rollback automatically only when safe; otherwise stop with restore instructions.

## Implementation Tests

- Dry run writes a plan and does not mutate data.
- Confirmation refuses missing, expired, or mismatched plans.
- Backup failure prevents destructive operations.
- Checksum/manifest verification is required before reset.
- Upgrade code path cannot invoke `reinit-data` or destructive helpers.
- Import scope includes only provider/LLM and connected remote settings.
- Excluded data classes are absent after reinit.
- Restore works from a failed partial run.
- Audit output redacts secrets.
