# Reinit Data Failsafe Design

Status: dry-run planning, backup-gated reset, operational provider/device
preserve, and verified restore are implemented in `scripts/install-server.mjs`.
The default confirmation flow creates a verified full backup, exports a narrow
operational preserve archive, moves the existing data root aside, imports only
the operational archive, and starts HanaAgent with a fresh test state.

## Goal

Define `install-server reinit-data` as a separate, explicit, recoverable reset/import workflow for first-run or new-user testing. It must never be part of `install-server upgrade`.

The implemented default result is a fresh workspace that preserves only the
narrow operational settings required to reconnect a test client:

- Providers/LLM settings.
- Access & Devices / Connected Remote Hana settings.

Persona, agents, memories, sessions, workspaces, plugins, user files, and broad customizations are not preserved by default.

## Command Shape

```sh
install-server backup --output <backup-path>
install-server reinit-data --dry-run
install-server reinit-data --dry-run --reset-pairing
install-server reinit-data --confirm <plan-id>
install-server reinit-data --restore <backup-path>
install-server reinit-data --restore latest-full-state
```

Required behavior:

- `--dry-run` is the default mode when neither `--confirm` nor `--restore` is present.
- Default dry run preserves provider/model setup and Connected Remote Hana bootstrap.
- `--reset-pairing` requests a fully fresh root and does not preserve provider, identity, device credential, or network bootstrap state.
- `--confirm <plan-id>` must reference a plan generated by a prior dry run.
- Confirmation must be impossible if the plan has expired, the data root changed, or the backup cannot be verified.
- `--restore <backup-path>` restores from a verified backup and does not run reset/import steps.
- `--restore latest-full-state` restores the newest full backup created immediately before a default operational Path B reinit.
- `backup --output <backup-path>` creates and verifies the same data-root archive used by confirmation.

Current implementation boundary:

- `reinit-data --dry-run` writes a plan file and does not mutate application data.
- `reinit-data --confirm` loads a non-expired dry-run plan, refuses data-root mismatch, stops `hanaagent`, creates and verifies a backup, exports preserved operational config when enabled, moves the current data root aside, imports the operational config, starts `hanaagent`, runs a systemd health check, and writes an audit file.
- `reinit-data --restore` verifies the backup archive, stops `hanaagent`, moves the current data root aside, restores the archive, fixes ownership, starts `hanaagent`, runs a systemd health check, and writes an audit file.

## sg01 Quick Start

CWD: `/root/openhanako`

Start every run by syncing the fork checkout and setting the command variables:

```bash
cd /root/openhanako
git pull --ff-only
git status --short --branch
export DATA=/root/.hanako PLAN_DIR=/var/lib/hanaagent/reinit-plans BACKUP_DIR=/opt/hanaagent/backups STAMP=$(date -u +%Y%m%dT%H%M%SZ) IS="node scripts/install-server.mjs"
```

On sg01, `git status --short --branch` should show `## dev...origin/dev`.
An untracked `?? bun.lock` is local checkout noise and is not part of this flow.

### Path A: Full Clear, Then Restore Pre-Path-B State

Use this after Path B when testing the full-clear/new-pairing loop, but you
want to return to the full state that existed before Path B. Do not take a new
manual backup after Path B for this case; that would capture the already-cleaned
Path B state. Restore the selector alias instead.

```bash
$IS reinit-data --dry-run --reset-pairing --data-root "$DATA" --plan-dir "$PLAN_DIR" | tee "/root/hana-reinit-fullclear-plan-$STAMP.json"
export PLAN_ID=$(node -pe 'require(process.argv[1]).planId' "/root/hana-reinit-fullclear-plan-$STAMP.json")
$IS reinit-data --confirm "$PLAN_ID" --data-root "$DATA" --plan-dir "$PLAN_DIR" | tee "/root/hana-reinit-fullclear-confirm-$STAMP.json"
$IS reinit-data --restore latest-full-state --data-root "$DATA" | tee "/root/hana-restore-full-state-$STAMP.json"
```

For a standalone Path A drill that has not run Path B first, use an explicit
manual backup path:

```bash
$IS backup --output "$BACKUP_DIR/manual-before-clear-$STAMP.tar.gz" --data-root "$DATA" | tee "/root/hana-backup-$STAMP.json"
export BACKUP=$(node -pe 'require(process.argv[1]).backupPath' "/root/hana-backup-$STAMP.json")
$IS reinit-data --dry-run --reset-pairing --data-root "$DATA" --plan-dir "$PLAN_DIR" | tee "/root/hana-reinit-fullclear-plan-$STAMP.json"
export PLAN_ID=$(node -pe 'require(process.argv[1]).planId' "/root/hana-reinit-fullclear-plan-$STAMP.json")
$IS reinit-data --confirm "$PLAN_ID" --data-root "$DATA" --plan-dir "$PLAN_DIR" | tee "/root/hana-reinit-fullclear-confirm-$STAMP.json"
$IS reinit-data --restore "$BACKUP" --data-root "$DATA" | tee "/root/hana-restore-explicit-$STAMP.json"
```

### Path B: Fresh Test State, Keep Provider/Connection

Use this for the day-one test state: sessions, memories, uploads, workspaces,
and broad user state are cleared, while provider/model setup and Connected
Remote Hana bootstrap are restored automatically. The confirm command still
creates and verifies a full backup before moving data aside.
That full backup is tagged as `latest-full-state`, so a later Path A can restore
the pre-Path-B state without accidentally selecting a backup taken after Path B.

```bash
$IS reinit-data --dry-run --data-root "$DATA" --plan-dir "$PLAN_DIR" | tee "/root/hana-reinit-operational-plan-$STAMP.json"
export PLAN_ID=$(node -pe 'require(process.argv[1]).planId' "/root/hana-reinit-operational-plan-$STAMP.json")
$IS reinit-data --confirm "$PLAN_ID" --data-root "$DATA" --plan-dir "$PLAN_DIR" | tee "/root/hana-reinit-operational-confirm-$STAMP.json"
```

Check after either path:

```bash
systemctl is-active hanaagent && ss -tlnp | grep ':14500'
```

## Data Root

The implementation must resolve the active Hana data root the same way the server does:

- `HANA_HOME` when set.
- Otherwise the application default, currently `~/.hanako` for CLI/server flows.

The command must print the resolved data root in every dry run and confirmation prompt. If the server is running, status output should identify the active process and service before planning destructive steps.

## Built-In Mechanism Discovery

Before implementing custom file deletion, the implementation work item must inspect and prefer built-in Hana mechanisms:

- CLI commands.
- HTTP/server routes.
- App export/import flows.
- Reset/onboarding flows.
- Existing auth/preferences/provider stores.

If a built-in export/import or reset path exists and can be tested, use it. Custom deletion is allowed only after documenting why no suitable built-in path exists for the exact data class.

## Dry-Run Plan

`install-server reinit-data --dry-run` creates a plan file under an operator-visible plan directory, for example:

```text
/var/lib/hanaagent/reinit-plans/<timestamp>-<short-hash>.json
```

The plan must include:

- Plan id.
- Created time and expiry.
- Resolved data root.
- Service name and current service state.
- Files/directories to back up.
- Files/directories that would be removed or replaced.
- Settings to export/import, with explicit field categories.
- Backup destination.
- Restore command.
- Health checks to run after reset.
- A "not preserved" section naming excluded classes.

The dry run must not mutate application data except for writing the plan file itself.

## Backup Gate

No destructive step may run before a verified backup exists.

Backup requirements:

1. Stop or quiesce the service when needed to get a consistent snapshot.
2. Archive the entire resolved data root.
3. Include release/service metadata and the dry-run plan in the manifest where available.
4. Write a manifest with file list, sizes, and SHA-256 checksums.
5. Verify the archive by reading it back and checking the manifest.
6. Print the backup path and restore command.

If backup creation or verification fails, the command exits before reset. It must not partially clear data.

Current implementation creates a gzip tar archive of the resolved data root,
writes a JSON manifest with the archive SHA-256 and tar listing, verifies the
archive with `tar -tzf` and `sha256sum`, and attempts to restart `hanaagent` if
backup fails after the service was stopped.
Backups created by default operational Path B confirmations are marked with
`restoreClass: full-state-before-operational-reinit`; `reinit-data --restore
latest-full-state` resolves the newest matching manifest and ignores later
manual backups or `--reset-pairing` confirmation backups.

## Operational Preserve Scope

The export/import scope is narrow by default. `--reset-pairing` disables this
preserve archive and produces a fully fresh root.
At execution time, only entries that currently exist under the data root are
archived; missing optional files do not fail the reset. If no operational
entries exist, the preserve manifest records a skipped preserve import and the
fresh-root reset still proceeds after the verified full backup.

Preserved provider/model setup:

- `auth.json`
- `added-models.yaml`
- `models.json`
- `models-cache.json`
- `provider-catalog.json`
- `provider-plugins/`
- `user/preferences.json`

Preserved connection/bootstrap setup:

- `server-network.json`
- `server-node.json`
- `users.json`
- `studios.json`
- `studio-mounts.json`
- `devices.json`
- `device-credentials.json`
- `pairing-sessions.json`
- `local-user-auth.json`
- `security/grants.json`

`server-info.json` is not preserved because it is runtime PID/token state and
must be regenerated by the restarted service.
`security/execution-leases.json` is not preserved because execution leases are
transient write grants, not durable provider or connection bootstrap.

Excluded by default:

- Persona.
- Agents.
- Memories.
- Session history.
- Workspaces and desk content.
- Uploaded files and generated media.
- Plugins and plugin state.
- `server-info.json`
- `security/execution-leases.json`

## Reset Sequence

The target confirmation flow is:

1. Load and validate the dry-run plan.
2. Re-resolve the data root and refuse if it differs from the plan.
3. Verify the plan is not expired.
4. Stop or quiesce `hanaagent`.
5. Create and verify the failsafe backup.
6. Export the operational preserve archive unless the plan was created with `--reset-pairing`.
7. Move the current data root aside.
8. Import only the preserved operational archive when enabled.
9. Start `hanaagent`.
10. Run health checks.
11. Write an audit report next to the backup and plan.

If health checks fail after reset, the command prints the restore command and may auto-restore only if it can prove no new user data was created after reset. Otherwise it stops and leaves the verified backup untouched for operator restore.

Current implemented confirmation flow:

1. Load and validate the dry-run plan.
2. Re-resolve the data root and refuse if it differs from the plan.
3. Verify the plan is not expired.
4. Stop `hanaagent`.
5. Create and verify the failsafe backup.
6. Export the operational preserve archive when enabled.
7. Move the current data root aside with a timestamp.
8. Import the preserved operational archive when enabled.
9. Start `hanaagent`.
10. Run `systemctl is-active --quiet hanaagent`.
11. Write a redacted audit report.

## Restore Sequence

`install-server reinit-data --restore <backup-path>` or
`install-server reinit-data --restore latest-full-state`:

1. Resolves `latest-full-state` to the newest operational full-state backup when the selector alias is used.
2. Verifies the backup archive and manifest.
3. Stops or quiesces `hanaagent`.
4. Moves the current data root aside with a timestamp; do not delete it immediately.
5. Restores the backup data root.
6. Fixes ownership and permissions.
7. Starts `hanaagent`.
8. Runs health checks.
9. Prints the restored version, data root, and retained aside path.

Restore must be available even if a previous reinit run failed halfway.

## Audit Output

Every confirmed run writes an audit file containing:

- Operator command.
- Plan id.
- Backup path.
- Data root.
- Exported categories.
- Removed/replaced paths.
- Health check result.
- Restore command.
- Start/end timestamps.

The audit output must avoid printing secret values. It may print key names, provider ids, model ids, and redacted credential presence.

## Failure Policy

- Missing plan id: show dry-run instructions and exit nonzero.
- Plan expired: require a fresh dry run.
- Data root mismatch: refuse to continue.
- Backup verification failure: refuse to reset.
- Export failure for allowed settings: refuse to reset unless the operator explicitly created the dry-run plan with `--reset-pairing`.
- Reset failure before import: print restore command.
- Import failure: keep backup and print both restore and retry instructions.
- Health check failure: rollback automatically only when safe; otherwise stop with restore instructions.

## Implementation Tests

- Dry run writes a plan and does not mutate data.
- Confirmation refuses missing, expired, or mismatched plans.
- Backup failure prevents destructive operations and attempts to restart the service.
- Checksum/manifest verification is required before reset.
- Upgrade code path cannot invoke `reinit-data` or destructive helpers.
- Restore verifies the backup before replacing the current data root.
- Shell operations stay inside explicit command allowlists and do not use `rm -rf`.
- Audit output redacts secrets.
- Default confirmation exports/imports operational config before service start.
- `--reset-pairing` skips operational preserve export/import.
